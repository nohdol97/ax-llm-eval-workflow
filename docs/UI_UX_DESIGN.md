## 용어집 (Glossary)

| 용어 | 정의 |
|------|------|
| **Experiment (실험)** | 특정 프롬프트/데이터셋/모델/평가 조합으로 실행되는 최상위 단위. 복수의 Run을 포함 |
| **Run** | 실험 내 1개 프롬프트 버전 × 1개 모델 조합의 실행 단위 |
| **Trace** | Langfuse의 단일 LLM 호출 기록. 아이템 하나당 하나의 trace |
| **Item (Dataset Item)** | 데이터셋의 개별 입력-기대출력 쌍 |
| **Evaluator** | 출력 품질을 측정하는 함수 (내장/LLM-Judge/Custom) |
| **Judge** | LLM-as-Judge — 다른 LLM이 출력을 평가하는 방식 |
| **Label Promotion (라벨 승격)** | 프롬프트 버전을 production/staging 라벨로 태깅 |
| **단일 테스트 (Playground)** | URL은 `/playground`이나 UI 표기는 "단일 테스트"로 통일 |
| **Compare Basket (비교 장바구니)** | 페이지 간 유지되는 비교 대상 실험 세트 (localStorage) |
| **Notification Inbox (알림 수신함)** | 완료/실패 알림을 복기하는 영구 저장 리스트 |
| **SSE** | Server-Sent Events. 단일 테스트 스트리밍 응답과 배치 실험 진행률에 사용 |
| **RBAC** | Role-Based Access Control. admin/user/viewer 3 역할 |
| **PII** | Personally Identifiable Information. 개인식별정보 |

## 문서 규칙

- **[Wireframe]**: ASCII/도식 기반 시각 레이아웃
- **[Spec]**: 구현 스펙 (표/규칙/수치)
- **[Guideline]**: 원칙/철학 (자유도 있음)
- **[ADR]**: Architecture Decision Record — 결정과 근거 기록
- **[Canonical]**: 해당 주제의 단일 진실 원본. 다른 섹션은 참조 관계

중복 주제가 여러 섹션에 걸쳐 있을 경우 각 섹션 헤더 근처에 Canonical 링크를 명시한다.

## 1. 참고 이미지 분석 — 현재 문제점

### 이미지 1: 실험 설정 화면
| 문제 | 상세 |
|------|------|
| 시각적 위계 부재 | 설정, 프롬프트, 파라미터 영역의 구분이 단순 border로만 처리되어 정보 계층이 불명확 |
| 균일한 여백 | 모든 요소의 간격이 동일하여 "AI가 배치한 듯한" 기계적 느낌 |
| 폼 레이아웃 | Temperature, Top P 등 슬라이더가 2열 그리드에 균등 배치 — 사용 빈도와 무관하게 같은 크기 |
| 다크 테마 안일함 | 순수 검정(#000) 배경에 회색 border만 사용, 깊이감 없음 |
| 탭 디자인 | "단일 테스트 / 실험 실행 / 데이터셋 업로드" 탭이 상단에 작게 배치되어 현재 맥락 인지 어려움 |
| 프로바이더 설정 | Azure/Gemini/Bedrock 설정이 접이식 아코디언으로 나열 — 사용자가 관리해야 할 영역과 실험 영역이 혼재 |

### 이미지 2: 실험 간 비교 대시보드
| 문제 | 상세 |
|------|------|
| 차트 3개 동일 크기 수평 나열 | Latency, Cost, Score가 같은 비중으로 배치되어 핵심 지표가 무엇인지 전달하지 못함 |
| 꺾은선 차트만 사용 | 3개 지표 모두 line chart — 데이터 특성에 맞지 않는 시각화 (cost는 bar가 적합) |
| 테이블 정보 밀도 부족 | Name 컬럼에 모델명이 잘려서 표시, Run Items/Latency/Cost 수치만 단순 나열 |
| 비교의 핵심 부재 | "어떤 Run이 왜 좋은가"에 대한 인사이트가 없음, 승자 표시 없음 |
| 색상 구분 | Run 간 색상 차이가 미미하여 차트에서 구분 어려움 |

### 이미지 3: 아이템별 상세 비교
| 문제 | 상세 |
|------|------|
| 카드 그리드의 정보 과밀 | 각 아이템 카드에 input/output/score/latency/cost가 작은 글씨로 빽빽하게 나열 |
| 비교 불가능한 레이아웃 | Run A와 Run B의 결과가 카드 내부에서 좌우로 나뉘지만, 시선 이동 경로가 불명확 |
| 우측 사이드바 차트 | 비교 차트가 사이드바에 밀려 작게 표시되어 가독성 저하 |
| 스코어 시각화 부재 | exact_match 0.0000 vs 1.0000이 텍스트로만 표시, pass/fail 직관성 없음 |
| 색상 코딩 없음 | 성공/실패/경고를 색상으로 구분하지 않아 스캔 불가 |

### 공통 문제: "AI가 만든 디자인"의 특징
1. **균일한 그리드**: 모든 요소가 같은 크기, 같은 간격 — 의도적 리듬감 없음
2. **기본 컴포넌트**: 브라우저 기본 폼 요소에 다크 스킨만 입힌 수준
3. **색상 체계 부재**: 의미 없는 회색 톤 변화, 브랜드 컬러 없음
4. **타이포그래피 단조로움**: 본문과 제목의 크기/굵기 차이만 있고 서체 특성 없음
5. **애니메이션 제로**: 상태 전환, 호버, 로딩에 움직임이 전혀 없음
6. **여백의 의도 부재**: 모든 padding/margin이 16px 또는 24px 단일값

## 2. 디자인 철학

### 2.1 핵심 원칙

**"도구는 사라지고, 워크플로우만 남는다"**

사용자가 "UI를 사용하고 있다"고 느끼지 않아야 한다. 프롬프트를 고치고, 실험을 돌리고, 결과를 판단하는 본질적 행위에만 집중할 수 있는 환경.

| 원칙 | 설명 | 안티패턴 |
|------|------|----------|
| 정보 위계 | 지금 이 순간 중요한 것만 눈에 들어온다 | 모든 정보를 같은 크기로 나열 |
| 점진적 노출 | 필요할 때만 세부 옵션이 드러난다 | 모든 설정을 한 화면에 배치 |
| 의견 있는 기본값 | 대부분의 경우 아무것도 건드리지 않아도 된다 | 빈 폼에서 시작 |
| 맥락적 피드백 | 결과가 좋은지 나쁜지 즉시 알 수 있다 | 숫자만 나열 |
| 비교는 나란히 | 비교 대상은 항상 시선 이동 최소화 배치 | 페이지 전환 후 기억에 의존 |

### 2.2 "AI가 만든 디자인"을 피하는 구체적 방법

| AI 디자인 특징 | 우리의 대안 |
|---------------|-----------|
| 균일한 12-column 그리드 | 콘텐츠 특성에 따른 비대칭 레이아웃 (7:5, 8:4 비율) |
| 모든 카드 동일 높이 | 정보량에 따라 자연스럽게 다른 높이 (masonry 아님, 의도적 차등) |
| 16px 균일 여백 | 4/8/12/16/24/32/48px 스케일에서 관계에 따라 선택 |
| 순수 검정 배경 | 미세하게 다른 회색 레이어로 깊이감 (zinc-900, zinc-950 혼용) |
| 기본 sans-serif | Pretendard (한글), Inter (영문/숫자) — 서체에 개성 부여 |
| 테두리로만 영역 구분 | 배경색 차이 + 미세한 그림자로 레이어 표현 |
| 단색 아이콘 | Lucide 아이콘에 의미별 컬러 (성공=emerald, 실패=rose, 경고=amber) |
| 상태 변화 없음 | Framer Motion으로 의미 있는 전환 (결과 도착, 스코어 변동) |
| 모든 차트 동일 스타일 | 데이터 특성에 맞는 시각화 선택 (bar, area, sparkline, heatmap) |

## 3. 디자인 토큰

### 3.1 색상 시스템

```
배경 레이어 (다크 모드):
├── Layer 0 (페이지 배경):   zinc-950  #09090b
├── Layer 1 (카드/패널):     zinc-900  #18181b
├── Layer 2 (입력 필드):     zinc-800  #27272a
└── Layer 3 (호버/활성):     zinc-700  #3f3f46

텍스트 (WCAG AAA 7:1 이상 보장):
├── Primary:    zinc-50   #fafafa     — 제목, 핵심 수치 (15.4:1 on zinc-950)
├── Secondary:  zinc-300  #d4d4d8     — 보조 텍스트, 라벨 (9.5:1 on zinc-950)
├── Tertiary:   zinc-500  #71717a     — 비활성, 플레이스홀더 (4.6:1, 대형 텍스트만)
└── Inverse:    zinc-950  #09090b     — 밝은 배지 위 텍스트

시맨틱 컬러 (텍스트용은 밝은 변형, 배경용은 어두운 변형):
├── Accent (브랜드):  indigo-300  #a5b4fc   — 주요 액션, 선택 상태 (8.9:1)
├── Success:         emerald-400 #34d399   — 통과, 높은 스코어 (9.0:1)
├── Warning:         amber-400   #fbbf24   — 주의, 중간 스코어 (11.8:1)
├── Error:           rose-300    #fda4af   — 실패, 낮은 스코어 (8.5:1)
├── Info:            sky-400     #38bdf8   — 정보, 링크 (8.6:1)
└── Neutral:         zinc-500    #71717a   — 비활성, 보류

스코어 배지 색상 (배경 + 텍스트 조합):
├── 높은 스코어:  emerald-900 배경 + emerald-300 텍스트 (7.8:1)
├── 중간 스코어:  amber-900 배경 + amber-300 텍스트 (8.2:1)
└── 낮은 스코어:  rose-900 배경 + rose-300 텍스트 (7.1:1)

스코어 그라데이션 (0.0 → 1.0):
  0.0~0.3: rose-500    — 빨간 계열 (나쁨)
  0.3~0.7: amber-500   — 노란 계열 (보통)
  0.7~0.9: emerald-400 — 초록 계열 (좋음)
  0.9~1.0: emerald-300 — 밝은 초록 (우수)
```

### 3.2 타이포그래피

```
서체:
├── 한글: Pretendard Variable
├── 영문/숫자: Inter Variable
└── 코드/JSON: JetBrains Mono

크기 스케일 (rem):
├── Display:   2.25rem (36px)  — 페이지 제목 (거의 사용하지 않음)
├── Heading 1: 1.5rem  (24px)  — 섹션 제목
├── Heading 2: 1.25rem (20px)  — 카드 제목
├── Heading 3: 1.125rem(18px)  — 서브 섹션
├── Body:      0.875rem(14px)  — 기본 본문 (대시보드 기본 크기)
├── Small:     0.8125rem(13px) — 테이블 셀, 보조 텍스트
└── Caption:   0.75rem (12px)  — 타임스탬프, 메타데이터

굵기:
├── Regular (400): 본문, 설명
├── Medium (500):  라벨, 테이블 헤더
├── Semibold (600): 카드 제목, 수치 강조
└── Bold (700):    페이지 제목 (최소 사용)

수치 표기:
├── 큰 숫자 (KPI): Inter Semibold 24px, tabular-nums
├── 테이블 수치: JetBrains Mono Regular 13px, tabular-nums
└── 인라인 수치: Inter Medium 14px
```

### 3.3 여백 시스템

```
간격 스케일:
├── 2px:  인라인 요소 간 미세 간격
├── 4px:  아이콘과 라벨 사이
├── 8px:  관련 요소 그룹 내부
├── 12px: 폼 필드 간격
├── 16px: 카드 내부 패딩
├── 24px: 섹션 내 요소 간격
├── 32px: 섹션 간 간격
└── 48px: 주요 영역 간 분리

적용 규칙:
- 관련 있는 요소 → 가까이 (8~12px)
- 다른 그룹의 요소 → 멀리 (24~32px)
- "같은 것은 같은 간격, 다른 것은 다른 간격" — 균일 여백 금지
```

### 3.4 모서리/그림자

```
모서리 반경:
├── Small:  6px   — 버튼, 입력 필드, 배지
├── Medium: 8px   — 카드, 드롭다운
├── Large:  12px  — 모달, 토스트
└── Full:   9999px — 필 (pill) 모양 태그

그림자 (다크 모드):
├── Subtle:  0 1px 2px rgba(0,0,0,0.3)   — 카드 기본
├── Medium:  0 4px 12px rgba(0,0,0,0.4)  — 호버된 카드, 드롭다운
└── Strong:  0 8px 24px rgba(0,0,0,0.5)  — 모달, 플로팅 패널

깊이감 표현:
- 그림자보다 배경색 차이로 레이어를 구분하는 것이 다크 모드에서 효과적
- Layer 0(zinc-950) 위에 Layer 1(zinc-900) 카드를 올리면 자연스러운 깊이감
```

## 6. 컴포넌트 설계

### 6.1 스코어 표시 컴포넌트

스코어를 단순 숫자가 아닌 시각적으로 즉시 판단 가능한 형태로 표시한다.

```
스코어 배지 (Score Badge):
┌─────────┐
│ ● 0.92  │  ← emerald 배경 + 흰색 텍스트, 좌측에 색상 dot
└─────────┘

스코어 바 (Score Bar):
▓▓▓▓▓▓▓▓▓░  0.92  ← emerald 프로그레스 바 + 우측 수치

스코어 셀 (테이블 내):
● 0.92     ← 색상 dot + 수치 (컴팩트)
✕ 0.15     ← rose dot + 수치

색상 매핑:
  0.0~0.3  → rose    (dot + bar + badge 모두)
  0.3~0.7  → amber
  0.7~1.0  → emerald
```

## 12. SSE 에러/연결 복구 설계

### 12.1 SSE 에러 이벤트 처리

| SSE 이벤트 | UI 동작 |
|------------|---------|
| `event: error` (단일 테스트) | 결과 패널에 rose-900 배경 에러 메시지 표시 + [재시도] 버튼 |
| `event: error` (배치 실험, 아이템) | 해당 아이템 행에 ✕ 표시, 실험은 계속 진행 |
| `event: error` (배치 실험, 치명적) | 프로그레스 바 rose 색상 전환 + 에러 토스트 + [실패 재시도] 버튼 |

### 12.2 SSE 연결 끊김 복구

```
연결 끊김 감지 (EventSource onerror):
  1. 상단에 amber 배너: "연결이 끊어졌습니다. 재연결 중..."
  2. 자동 재연결 시도: 1초 → 2초 → 4초 → 8초 (exponential backoff, 최대 30초)
  3. 재연결 성공: 배너 제거 + "연결 복구됨" 토스트 (2초 후 사라짐)
  4. 30초 내 재연결 실패: 배너를 rose로 변경 + [수동 재연결] 버튼

배치 실험 연결 복구 시:
  - GET /experiments/{id} 로 현재 상태 재조회
  - 프로그레스 바 갱신 후 SSE 재구독
```

### 14.1 컴포넌트별 ARIA 설계

| 스코어 배지 | `aria-label="스코어 0.92, 우수"` (색상에만 의존하지 않음) |
| 토스트 알림 | `role="alert"`, `aria-live="polite"` |
| 모달 | `aria-modal="true"`, 포커스 트랩, Esc로 닫기 |

| 에러 메시지 | `role="alert"`, 입력 필드 `aria-describedby="error-id"` |

### 14.2 전역 접근성 (확장)

**키보드 네비게이션 상세**:
- 비교 테이블: `role="grid"`, Arrow Up/Down/Left/Right로 셀 이동, Home/End로 행 끝, PageUp/PageDown 페이지 이동, Enter로 아이템 상세 모달
- 위저드: `role="tablist"`의 스텝, Arrow Left/Right로 이전/다음, Enter로 다음 단계, Shift+Tab 역이동
- 드롭다운: Arrow Up/Down 옵션 이동, Home/End 처음/끝, type-ahead 검색, Enter 선택, Esc 닫기
- 차트: 데이터 포인트 Tab 이동, Arrow로 다음/이전 포인트, Enter로 아이템 상세

**포커스 관리**:
- 모달 열림: 첫 번째 입력 필드로 포커스. **파괴적 다이얼로그는 안전한 쪽(취소)에 기본 포커스**.
- 모달 닫힘: 트리거 요소로 포커스 복귀
- 위저드 단계 전환: 다음 스텝의 제목(`h2`, `tabindex="-1"`)으로 이동
- 폼 검증 실패: 첫 에러 필드로 스크롤 + 포커스 (scrollIntoView block: 'center')
- Tab 트랩: 모달 내부에서 Tab 순환 (마지막 → 첫 번째)

**전역 규칙**:
- `outline: none` 금지. 모든 인터랙티브 요소에 `*:focus-visible { outline: 2px solid var(--indigo-300); outline-offset: 2px; }`
- `<html lang="ko">`, 영문 모델명은 인라인 `<span lang="en">`
- Skip Nav 대상: `<main id="content">`
- `<fieldset>` + `<legend>`: 위저드 모델 그룹, 변수 그룹, 평가 함수 그룹, 컬럼 매핑

**폼 접근성**:
- 모든 input에 `<label for="id">` 또는 wrapping label
- 에러 메시지: `aria-describedby="error-id"`, `aria-invalid="true"`
- 필수 필드: `aria-required="true"` + "(필수)" 텍스트 병기 (별표만 금지)
- placeholder는 라벨 대체 불가 (별도 label 필수)
- 로그인 폼: `autocomplete="username"`, `autocomplete="current-password"`

**텍스트/가독성 (WCAG 1.4.12)**:
- 200% 확대 시 가로 스크롤/콘텐츠 잘림 없음
- 줄간격 최소 1.5, 문단 간격 최소 2배, 자간 0.12배, 단어 간격 0.16배
- 최대 행 길이 80ch (본문)
- 링크는 밑줄 또는 2.5:1 이상 대비 + 색상 외 식별자

**모션/애니메이션**:
- `prefers-reduced-motion: reduce` 존중: 모든 전환 애니메이션 0.01ms로 단축 (즉시 표시)
- 무한 스피너 금지: 5초 이상 지속 시 "계속 대기..." 텍스트 + [취소] 버튼 제공
- 자동 재생/블링크 금지 (3Hz 이하)

**상태 메시지 (WCAG 4.1.3)**:
- 토스트: `role="status"` (정보/성공) 또는 `role="alert"` (에러)
- 검색 결과 갱신: "총 {N}건 찾음" live region
- 저장/삭제 완료: live region으로 자동 안내
- SSE 스트리밍: 섹션 31 참조

## 16. 에러 상태 UI 설계 [Canonical]

> **Canonical**: 에러 코드별 사용자 메시지/UI 표현의 단일 진실 원본. 섹션 12는 SSE 연결 에러 전용, 섹션 19는 폼 입력 검증 전용.



### 16.1 에러 유형별 표현 방식

```
에러 표현 3단계:
├── 인라인 에러: 폼 필드 바로 아래 (rose-300 텍스트, 필드 border rose)
├── 토스트 에러: 우하단 토스트 (작업 중 발생한 비치명적 에러)
└── 전체 화면 에러: 페이지 전체 차지 (접근 불가, 인증 실패 등)
```

### 19.2 데이터셋 매핑 검증


## 25. 파괴적 액션 공통 규칙

### 25.1 확인 다이얼로그 규칙

- **기본 포커스**: 안전한 선택지(취소/계속 실행)에 기본 포커스
- **backdrop 클릭**: 닫기 **비허용** (실수 방지)
- **Esc 키**: 닫기 허용 (취소와 동일 효과)
- **처리 중 상태**: 확인 버튼 → 스피너 + disabled, 모달 자체도 닫기 불가
- **처리 완료 후**: 성공 시 자동 닫기 + 토스트, 실패 시 에러 표시 + 재시도 버튼

### 25.2 되돌릴 수 없는 작업 목록

| 작업 | 확인 방식 | 복구 가능 여부 |
|------|---------|-------------|
| 실험 삭제 | 이름 입력 필수 | 불가 (Langfuse trace는 유지) |
| 데이터셋 삭제 | 이름 입력 필수 | 불가 |
| 실험 취소 | "계속 실행" 기본 포커스 | 불가 |
| 프롬프트 라벨 승격 | 단순 확인 | 이전 버전으로 재승격 가능 |
| 커스텀 Judge 프롬프트 초기화 | 단순 확인 | 불가 (사용자 입력 손실) |
| 템플릿 덮어쓰기 | 동일 이름 시 충돌 경고 | localStorage 덮어씀 |

### 25.3 런타임 권한 변경 / 세션 중 JWT 만료

- **JWT 만료 (작업 중)**: 401 수신 시 현재 폼 상태를 `sessionStorage`에 "복구 키"로 저장 → 로그인 페이지로 리다이렉트 → 재로그인 성공 시 원래 페이지로 복귀 + "작업을 복구했습니다" 토스트
- **세션 중 권한 강등** (admin → user): 다음 API 요청에서 403 수신 시 UI 즉시 재렌더링 (버튼 숨김), "권한이 변경되었습니다" amber 배너 표시
- **다중 탭 상태 동기화**: `storage` 이벤트로 비교 장바구니/알림 수신함 unread 수를 탭 간 실시간 반영

### 25.4 의도치 않은 이탈 방지

- **업로드 중 페이지 이탈**: `beforeunload` 경고 "업로드가 진행 중입니다. 페이지를 떠나면 중단됩니다."
- **위저드 진행 중 뒤로가기**: "입력한 내용이 사라집니다. 계속하시겠습니까?" 확인
- **단일 테스트 스트리밍 중 탭 닫기**: 스트리밍 자체는 서버에서 계속 기록됨을 안내
- **배치 실험 실행 중 탭 닫기**: 실험은 백그라운드로 계속 실행됨을 안내
